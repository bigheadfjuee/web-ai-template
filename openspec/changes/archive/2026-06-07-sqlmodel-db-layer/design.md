## Context

The backend currently maintains two parallel class hierarchies for the `users` domain: a SQLAlchemy ORM `User` class (`app/models/user.py`) and three Pydantic classes (`UserCreate`, `UserUpdate`, `UserResponse` in `app/schemas/user.py`). The ORM class uses `sqlalchemy.dialects.postgresql.UUID` and `server_default=text("gen_random_uuid()")` — both PostgreSQL-specific. The CRUD router imports from both hierarchies.

SQLModel (by the FastAPI author) wraps SQLAlchemy 2 + Pydantic v2 and allows a single class to serve as both ORM table and Pydantic schema, eliminating the duplication and standardizing around one library surface.

## Goals / Non-Goals

**Goals:**
- Replace `sqlalchemy.dialects.postgresql.UUID` with `sqlalchemy.Uuid` (dialect-agnostic, supported by both PG and MSSQL via SQLAlchemy's type system).
- Replace `server_default=text("gen_random_uuid()")` with client-side `default_factory=uuid.uuid4` so no database function is called.
- Replace `server_default=text("now()")` for timestamps with Python-side `default_factory=lambda: datetime.now(tz=timezone.utc)` so no dialect-specific server function is used.
- Unify ORM + Pydantic schemas into SQLModel classes, reducing the number of files an implementer must touch when adding a new field.
- Keep the Alembic migration workflow (not replaced by `SQLModel.metadata.create_all`).
- Keep all REST endpoint contracts unchanged (same paths, status codes, response field names).

**Non-Goals:**
- No MSSQL driver installation or MSSQL Compose service.
- No changes to the frontend, nginx, or docker-compose.
- No new database migrations — `0001_create_users_table` is updated in place (development-only data).
- No change to the five REST endpoint paths or their status codes.
- No change to the `GET /api/health` endpoint.

## Decisions

### SQLModel table model replaces SQLAlchemy DeclarativeBase model

`User` in `backend/app/models/user.py` becomes `class User(SQLModel, table=True)` with `__tablename__ = "users"`. SQLModel integrates with the same `AsyncSession` and session factory from `sqlalchemy.ext.asyncio`. The `Base = DeclarativeBase()` in `backend/app/db.py` is replaced by `SQLModel.metadata` for Alembic's `target_metadata`.

`backend/app/db.py` removes `class Base(DeclarativeBase)` and instead uses `from sqlmodel import SQLModel` — `SQLModel.metadata` is passed to Alembic's `env.py` as `target_metadata`.

### Dialect-agnostic UUID primary key

`id` is declared as `Field(default_factory=uuid.uuid4, primary_key=True)` with `sa_column=Column(sa.Uuid, primary_key=True, default=uuid.uuid4)`. Using `sa.Uuid` (SQLAlchemy 2 generic UUID type) instead of `sqlalchemy.dialects.postgresql.UUID(as_uuid=True)` means the same column definition works on PostgreSQL, MSSQL, and SQLite.

UUID is generated by Python before the INSERT statement — no `gen_random_uuid()` SQL call, no extension dependency.

Alternative: keep PostgreSQL UUID dialect type and rely on Alembic render rendering the correct DDL per dialect. Rejected because it still imports from `sqlalchemy.dialects.postgresql`, signalling dialect coupling and requiring a conditional render_as_batch or dialect check in migrations.

### Dialect-agnostic timestamps with Python-side defaults

`created_at` and `updated_at` use `sa_column=Column(sa.DateTime(timezone=True))` with `default_factory=lambda: datetime.now(tz=timezone.utc)` on `created_at`, and the same default on `updated_at` (overwritten explicitly on UPDATE in the router). No `server_default=text("now()")` — the value is always provided by Python.

`sa.DateTime(timezone=True)` renders as `TIMESTAMPTZ` on PostgreSQL and `DATETIMEOFFSET` on MSSQL — both timezone-aware, both handled by SQLAlchemy's type system.

### SQLModel schema classes replace Pydantic BaseModel classes

`backend/app/schemas/user.py` is rewritten with SQLModel-native schemas:
- `UserBase(SQLModel)` — shared fields `username: str` and `email: EmailStr` (not a table)
- `UserCreate(UserBase)` — no additional fields; used for POST body
- `UserUpdate(SQLModel)` — `username: str | None = None`, `email: EmailStr | None = None` (partial update; NOT inheriting UserBase to keep all fields optional)
- `UserPublic(UserBase)` — `id: uuid.UUID`, `created_at: datetime`, `updated_at: datetime`; `model_config = ConfigDict(from_attributes=True)`

`UserResponse` is renamed to `UserPublic` (SQLModel convention). The router is updated to import `UserPublic` instead of `UserResponse`. The REST response shape is identical — only the Python class name changes.

### Alembic migration updated in place (no new revision)

The existing `0001_create_users_table.py` is updated to use `sa.Uuid` and Python-side defaults (no `server_default`). Since this is a development template and the volume will be recreated (`docker compose down -v && docker compose up --build`), no new migration is needed. The `downgrade()` function remains identical (drops the table).

Alternative: create a new migration `0002` that alters the `id` column type. Rejected because no production data exists and the column type change from dialect-specific UUID to `sa.Uuid` produces identical DDL on PostgreSQL anyway — the migration output is unchanged on PG; it is only the Python import that differs.

## Implementation Contract

**Observable behaviors (acceptance):**

- After `docker compose down -v && docker compose up --build`, the backend starts, `alembic upgrade head` succeeds, and all seven curl assertions from the previous change pass identically: list → `200 []`, create → `201` with UUID `id`, get → `200`, PUT → `200` with updated `updated_at`, DELETE → `204`, second DELETE → `404`, duplicate → `409`.
- `uv run python -c "from app.models.user import User; print(User.__tablename__)"` prints `users`.
- `uv run python -c "from app.schemas.user import UserPublic; print(list(UserPublic.model_fields.keys()))"` prints all five field names (`id`, `username`, `email`, `created_at`, `updated_at`).
- `from sqlalchemy.dialects.postgresql import UUID` MUST NOT appear anywhere in `backend/app/` after this change.
- `gen_random_uuid` MUST NOT appear anywhere in `backend/` after this change.
- `npm run build` from `frontend/` continues to exit 0 (no frontend files changed).

**Interfaces / data shapes:**

- All five `/api/users` endpoints retain identical paths, HTTP methods, status codes, and JSON field names. `UserResponse` → `UserPublic` is a Python-only rename; the serialized JSON is unchanged.
- `GET /api/users?skip=0&limit=100` → `200` `[{id, username, email, created_at, updated_at}, ...]`
- `POST /api/users` → `201` `{id, username, email, created_at, updated_at}`
- `PUT /api/users/{id}` → `200` same shape, `updated_at` refreshed
- `DELETE /api/users/{id}` → `204` no body
- `GET /api/health` → `200` `{"status":"ok"}` unchanged

**Failure modes:**

- If `SQLModel.metadata` is not passed to Alembic's `env.py` as `target_metadata`, `alembic upgrade head` will see no tables and do nothing; verify with `docker compose logs backend` showing `Running upgrade -> 0001`.
- If client-side UUID default is missing, inserts without an explicit `id` will fail with a NOT NULL constraint error; verify with the POST curl.
- IntegrityError → 409 mapping in the router MUST still catch `sqlalchemy.exc.IntegrityError` (SQLModel raises the same exceptions as SQLAlchemy).

**Acceptance verification:**

- Automated: `uv run python -c "from app.models.user import User; ..."` import checks pass (no dialect imports).
- Manual: `docker compose down -v && docker compose up --build` + seven-step curl sequence all pass.
- Grep: `grep -r "dialects.postgresql" backend/app/` returns no matches.
- Grep: `grep -r "gen_random_uuid" backend/` returns no matches.

**Scope boundaries:**

- In scope: `backend/app/db.py`, `backend/app/models/user.py`, `backend/app/schemas/user.py`, `backend/app/routers/users.py`, `backend/alembic/versions/0001_create_users_table.py`, `backend/alembic/env.py`, `backend/pyproject.toml`, `backend/uv.lock`.
- Out of scope: everything in `frontend/`, `nginx/`, `docker-compose.yml`, `backend/app/main.py`, `backend/app/config.py`.

## Risks / Trade-offs

- [SQLModel's async session support requires `sqlmodel.ext.asyncio.session.AsyncSession` or `sqlalchemy.ext.asyncio.AsyncSession`] → Both work; the router continues using `sqlalchemy.ext.asyncio.AsyncSession` from `get_db` with no change needed — SQLModel table models are fully compatible with raw SQLAlchemy sessions.
- [Alembic `env.py` `target_metadata` must switch from `Base.metadata` to `SQLModel.metadata`] → If missed, autogenerate sees no models and migrations become empty. Covered by the grep/log acceptance check.
- [Client-side UUID generation means the `id` column has no `server_default`] → Any INSERT that bypasses the ORM (e.g., raw SQL seed scripts) must supply the UUID explicitly. Acceptable for a template scaffold.
- [`updated_at` is no longer set by a database trigger] → Already handled in the router (`user.updated_at = datetime.now(tz=timezone.utc)` on PUT); no behavior change.
